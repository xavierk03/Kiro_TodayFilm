## 1. 제품 개요와 비전

### 1.1 한 줄 소개

> **사진과 AI 대화를 통해 하루를 돌아보고, 시간이 지나면 과거와 현재의 기억을 연결해 주는 AI 회고 서비스**

### 1.2 서비스 개요

오늘의 필름은 하루를 마무리하며 오늘의 사진을 선택하고, AI와 짧은 대화를 나누며 하루를 돌아보는 서비스입니다. AI는 사용자의 감정을 대신 판단하거나 분석하지 않습니다. 사진과 사용자의 답변을 단서로 질문을 이어가며 사용자가 스스로 하루를 돌아보도록 돕습니다.

회고가 끝나면 사진과 대화를 바탕으로 제목, 한 줄 회고, 기억 문장, 필름 카드가 생성됩니다. 기록이 쌓이면 과거의 고민·목표·관계·기대와 현재의 순간을 연결하여 사용자가 시간에 따른 변화를 발견하도록 합니다.

### 1.3 기획 배경

- 사진은 장면을 남기지만 왜 찍었는지, 무슨 생각을 했는지까지 남기지 못합니다.
- 일기는 생각을 남길 수 있지만 긴 글을 직접 작성해야 해 지속하기 어렵습니다.
- AI 기록 서비스가 감정을 분류하거나 결론 내리면 사용자의 실제 경험과 어긋날 수 있습니다.
- 날짜별로 쌓인 기록은 사용자가 직접 찾지 않으면 과거와 현재의 변화를 보여주지 못합니다.

오늘의 필름은 다음 질문에서 출발합니다.

> **AI가 나의 하루를 대신 분석하는 것이 아니라, 내가 스스로 하루를 돌아볼 수 있도록 도와줄 수는 없을까?**

### 1.4 문제 정의

| ID | 문제 | 설명 |
|---|---|---|
| P1 | 사진은 남지만 그날의 이야기는 사라진다 | 사진을 찍은 이유와 당시의 생각은 기록되지 않아 시간이 지나면 개인적인 기억이 흐려진다. |
| P2 | 기록을 꾸준히 이어가기 어렵다 | 긴 글을 직접 써야 하는 부담과 평범한 하루에 무엇을 쓸지 모르는 막막함 때문에 기록을 중단한다. |
| P3 | 기록은 쌓이지만 서로 연결되지 않는다 | 과거의 고민·목표가 지금 어떻게 달라졌는지 사용자가 직접 이전 기록을 찾지 않으면 발견하기 어렵다. |

### 1.5 프로젝트 목표

| ID | 목표 | 대응 문제 |
|---|---|---|
| G1 | 긴 글 없이 사진과 짧은 대화로 기록의 부담을 낮춘다. | P2 |
| G2 | AI가 감정을 판단하는 대신 질문으로 사용자의 자기 회고를 돕는다. | P1 |
| G3 | 과거와 현재의 연결점을 제시해 시간에 따른 변화를 발견하게 한다. | P3 |
| G4 | 회고를 시각적인 필름으로 완성해 지속하고 싶은 기록 경험을 만든다. | P2 |

### 1.6 제품 성공 지표

MVP 출시 후 아래 지표를 측정합니다. 초기 목표치는 사용자 테스트 결과를 바탕으로 조정합니다.

| ID | 지표 | 측정 방법 |
|---|---|---|
| KPI-1 | 회고 완료율 | 사진 선택을 완료한 세션 중 결과 생성까지 완료한 비율 |
| KPI-2 | 회고 소요 시간 | 사진 선택 완료부터 결과 확인까지의 중앙값과 p95 |
| KPI-3 | 재사용률 | 첫 기록 후 7일·30일 안에 다시 기록한 사용자 비율 |
| KPI-4 | 질문 건너뛰기율 | 전체 질문 중 건너뛴 질문의 비율 |
| KPI-5 | 결과 수정률 | 생성된 제목·한 줄 회고를 사용자가 수정한 비율 |
| KPI-6 | AI 원칙 위반률 | 평가 시나리오에서 감정 단정·사실 추측이 발생한 비율 |
| KPI-7 `[확장]` | 과거 연결 수용률 | 제시된 과거 기록 중 사용자가 반응하거나 열람한 비율 |

### 1.7 차별점

- **판단하지 않는 AI:** 감정 단정과 사진 속 사실 추측을 금지하고 질문으로 회고를 돕습니다.
- **사진 기반 시작점:** 빈 화면 대신 오늘 찍은 사진에서 대화를 시작합니다.
- **시간 축 연결:** 관리 가능한 기억 포인트를 통해 과거와 현재의 변화를 발견하게 합니다.

### 2.1 사용자 유형

| ID | 사용자 유형 | 주요 요구 |
|---|---|---|
| T1 | 사진은 자주 찍지만 긴 일기는 부담스러운 사람 | 짧고 쉽게 시작할 수 있는 기록 |
| T2 | 평범한 일상도 기록으로 남기고 싶은 사람 | 특별한 사건이 없어도 이어갈 수 있는 질문 |
| T3 | 과거 기록에서 자신의 변화를 발견하고 싶은 사람 | 과거와 현재를 신뢰할 수 있게 연결하는 경험 |

### 2.2 작업용 페르소나

> **지현, 29세 직장인** — 사진은 자주 찍지만 일기는 오래 이어가지 못합니다. “행복해 보여요”처럼 감정을 단정하는 AI 표현을 불편해합니다. 하루 2~3분 동안 사진 한 장에서 시작해 가볍게 돌아보고, 시간이 지난 뒤 그때의 자신과 현재를 비교하고 싶어 합니다.

이 페르소나는 T1~T3의 공통 요구를 검증하기 위한 작업용 모델이며, 사용자 인터뷰 결과에 따라 수정할 수 있습니다.

## 3. 범위

### 3.1 In Scope `[MVP]`

- 이메일/비밀번호 및 Google/Apple 로그인
- 사용자 현지 날짜 기준 하루 1기록
- 오늘 사진 추천 및 갤러리에서 사진 1~3장 선택
- 모든 기록에 대표 사진 1장 지정
- 이미지 리사이즈·압축·EXIF 위치정보 제거 후 업로드
- 시작 → 확장 → 마무리의 최대 3문 AI 회고
- 질문 건너뛰기, 중간 종료, 미완료 회고 이어쓰기
- 제목·한 줄 회고·기억 문장·필름 카드 1종 생성
- 대화 중 자동 저장과 사용자별 비공개 보관
- 월간 캘린더와 날짜별 기록 상세 화면
- 계정 및 기록 삭제

### 3.2 In Scope `[확장]`

- 사진 4~5장 선택
- 기록 패턴 기반 개인화 질문
- 기억 포인트 추출 및 pgvector 기반 과거 기록 연결
- 포스터 등 시각 결과물 다양화
- 연간 아카이브와 주제·키워드별 모아보기

### 3.3 Out of Scope

- 감정 점수화 및 감정 분석 대시보드
- 소셜 피드와 공개 커뮤니티
- 대화 원문 공유
- 실시간 공동 편집
- 웹 클라이언트
- 의료·심리 상담이나 진단을 대신하는 기능

### 3.4 데모 범위

과거 기록 연결은 데이터 누적이 필요하므로, 데모에서는 사전 준비한 과거 기록 1건으로 연결 시나리오를 시연할 수 있습니다. 데모 데이터임을 명시하며 실제 사용자 데이터와 분리합니다.

## 4. 기능 요구사항

### FR-0. 인증 및 계정 관리 `[MVP]`

- FR-0.1 사용자는 이메일/비밀번호 또는 Google/Apple 계정으로 가입·로그인할 수 있다.
- FR-0.2 사용자는 로그아웃하고 이메일 계정의 비밀번호를 재설정할 수 있다.
- FR-0.3 인증 세션이 만료되면 재인증을 안내하고 저장되지 않은 로컬 입력을 보호한다.
- FR-0.4 사용자는 계정과 자신이 생성한 사진·대화·필름 카드를 삭제할 수 있다.
- FR-0.5 사용자는 다른 사용자의 기록에 접근할 수 없다.

### FR-1. 사진 선택 및 정제 `[MVP]`

- FR-1.1 앱은 사용자 현지 날짜와 사진 촬영 메타데이터를 기준으로 오늘 촬영한 사진을 추천한다.
- FR-1.2 촬영일 메타데이터가 없으면 기기 미디어 생성일을 사용하고, 날짜를 확인할 수 없으면 일반 갤러리 선택만 제공한다.
- FR-1.3 사용자는 사진을 1~3장 선택할 수 있다. `[확장]` 최대 5장까지 허용한다.
- FR-1.4 선택 사진 수와 관계없이 대표 사진을 정확히 1장 지정한다. 첫 번째 선택 사진을 기본 대표로 설정하고 변경할 수 있다.
- FR-1.5 앱은 업로드 전에 이미지를 리사이즈·압축하고 EXIF 위치정보를 제거한다.
- FR-1.6 서버는 정제 여부와 허용 형식·크기를 검증하며, 정제되지 않은 원본을 영구 저장하지 않는다.
- FR-1.7 갤러리 권한이 거부되면 권한 필요 이유와 설정 이동 방법을 안내한다.

### FR-2. AI 회고 대화 `[MVP]`

- FR-2.1 Vision 모델은 대표 사진의 맥락을 분석해 개방형 첫 질문을 생성한다.
- FR-2.2 사진 맥락은 질문 생성에만 사용하며 분석 결과를 사실처럼 사용자에게 노출하지 않는다.
- FR-2.3 후속 질문은 직전 답변의 핵심 표현과 지금까지의 대화를 이어받는다.
- FR-2.4 대화는 시작 → 확장 → 마무리의 최대 3문 구조이며, 남은 질문 수를 표시한다.
- FR-2.5 사용자는 모든 질문을 건너뛰거나 언제든 대화를 종료할 수 있다.
- FR-2.6 2회 연속 건너뛰면 조기 종료를 제안한다.
- FR-2.7 감정 단정, 사진 속 사실 추측, 답변 유도 금지 원칙을 시스템 프롬프트와 평가 규칙으로 고정한다.
- FR-2.8 AI 호출 실패 시 재시도 후에도 복구되지 않으면 검증된 기본 질문 풀로 회고를 계속할 수 있다.
- FR-2.9 Gemini 호출은 인증된 서버 측 AI Gateway에서만 수행하며 클라이언트에 API 키를 노출하지 않는다.

### FR-3. AI 결과물 생성 `[MVP]`

- FR-3.1 결과 생성 입력은 정제 사진, 대표 사진 맥락, 저장된 대화이다.
- FR-3.2 결과는 오늘의 제목, 한 줄 회고, 기억 문장, 대화 전체, 필름 카드로 구성한다.
- FR-3.3 LLM은 다음 구조의 JSON을 반환한다.

```json
{
  "title": "오늘의 제목",
  "one_line": "한 줄 회고",
  "quote": "사용자 답변의 원문"
}
```

답변에서 적합한 기억 문장을 찾을 수 없는 경우 `quote` 값은 JSON `null`을 사용한다.

- FR-3.4 `quote`는 사용자 답변의 원문을 우선 사용하며 적합한 문장이 없으면 `null`을 허용한다.
- FR-3.5 AI는 대화에 없던 감정·사건·인물 관계를 추가하지 않는다.
- FR-3.6 사용자는 제목과 한 줄 회고를 수정할 수 있으며, 기억 문장의 출처 답변을 확인할 수 있다.
- FR-3.7 Flutter `RepaintBoundary`로 필름 카드 1종을 생성하고 정제 사진과 함께 저장한다.
- FR-3.8 구조화 출력 검증에 실패하면 제한 횟수만큼 재생성하고, 계속 실패하면 안전한 로컬 기본값으로 최소 기록을 만든다.

### FR-4. 기록 자동 저장 및 상태 관리 `[MVP]`

- FR-4.1 사진 선택을 완료하면 `draft` 상태의 Film을 생성한다.
- FR-4.2 질문과 답변은 각 상호작용이 끝날 때 자동 저장한다.
- FR-4.3 회고 종료 후 결과 생성 중에는 Film을 `generating` 상태로 전환한다.
- FR-4.4 결과 저장을 완료하면 Film을 `completed` 상태로 전환한다.
- FR-4.5 결과 생성에 실패하면 Film을 `failed` 상태로 표시하고 재시도 또는 최소 기록 완료를 제공한다.
- FR-4.6 당일 `draft` 기록에 재진입하면 “회고 이어하기”와 “현재 기록 보기”를 제공한다.
- FR-4.7 당일 `completed` 기록에 재진입하면 기록 상세 화면을 보여준다.
- FR-4.8 자동 저장 요청은 중복 실행되어도 동일한 결과를 유지하도록 멱등적으로 처리한다.
- FR-4.9 저장 단위는 기록 날짜·시간대·정제 사진·대화·제목·한 줄 회고·기억 문장·필름 카드이다.

### FR-5. 기록 아카이브 `[MVP]`

- FR-5.1 월간 캘린더에서 완료 기록이 있는 날짜에 대표 사진 썸네일을 표시한다.
- FR-5.2 날짜를 선택하면 정제 사진, 대화, AI 결과물, 필름 카드를 보여준다.
- FR-5.3 미완료 기록은 완료 기록과 시각적으로 구분하고 이어쓰기 동작을 제공한다.
- FR-5.4 사용자는 개별 기록을 삭제할 수 있다.

### FR-6. 과거 기록 연결 `[확장]`

- FR-6.1 회고 종료 시 사용자 표현에서 고민·목표·관계·기대 유형의 기억 포인트를 구조화해 저장한다.
- FR-6.2 새 기록의 답변과 대표 사진 맥락을 차단되지 않은 기존 기억 포인트와 비교한다.
- FR-6.3 의미 유사도와 주제 태그가 기준을 만족할 때만 회고당 최대 1회 연결을 제시한다.
- FR-6.4 연결은 마지막 질문을 시간 연결형 질문으로 대체하거나 결과 화면의 “그때의 기록” 카드로 제시한다.
- FR-6.5 사용자가 반응하지 않은 연결은 결과물에 포함하지 않는다.
- FR-6.6 사용자는 특정 기록을 “다시 꺼내지 않기”로 설정하거나 기억 포인트를 삭제할 수 있다.
- FR-6.7 연결 후보가 없거나 신뢰도가 낮으면 연결 기능을 노출하지 않는다.

### FR-7. 개인정보 및 데이터 관리 `[MVP]`

- FR-7.1 최초 AI 회고 전에 사진과 대화가 AI 처리를 위해 외부 모델 제공자에게 전송됨을 고지하고 동의를 받는다.
- FR-7.2 서버 저장 이미지는 D7의 정제 이미지만 허용한다.
- FR-7.3 사진, 대화, AI 결과는 기본 비공개이며 Supabase RLS로 사용자별 접근을 격리한다.
- FR-7.4 서명 URL 등 제한된 방식으로만 비공개 이미지를 조회한다.
- FR-7.5 계정 또는 기록 삭제 시 연결된 DB 행과 Storage 객체를 함께 삭제한다.
- FR-7.6 대화 원문은 공유 대상에 포함하지 않는다.

## 6. 유저 스토리와 수락 기준

### US-0. 가입과 로그인

> 사용자로서, 내 기록이 다른 사람과 섞이지 않도록 안전하게 계정에 로그인하고 싶다.

- WHEN 사용자가 지원 로그인 방식을 선택하면 THEN 시스템은 Supabase Auth를 통해 인증한다.
- IF 인증 세션이 만료되면 THEN 시스템은 재로그인을 안내하고 작성 중인 로컬 입력을 보호한다.
- 시스템은 사용자 A가 사용자 B의 Film·Photo·Message를 조회하거나 변경하지 못하게 해야 한다.

### US-1. 회고 시작과 재진입

> 사용자로서, 홈에서 빠르게 오늘의 회고를 시작하거나 작성 중인 회고를 이어가고 싶다.

- WHEN 오늘 기록이 없고 사용자가 시작 버튼을 누르면 THEN 시스템은 사진 선택 화면을 보여준다.
- IF 오늘 기록이 `draft`이면 THEN 시스템은 “회고 이어하기”와 “현재 기록 보기”를 제공한다.
- IF 오늘 기록이 `completed`이면 THEN 시스템은 해당 기록 상세 화면을 보여준다.

### US-2. 사진 선택

> 사용자로서, 오늘 찍은 사진을 골라 회고의 단서로 삼고 싶다.

- WHEN 사진 선택 화면에 진입하면 THEN 오늘 촬영한 사진을 추천한다.
- 사용자는 MVP에서 1~3장을 선택할 수 있어야 한다.
- WHEN 사진을 선택하면 THEN 정확히 1장을 대표 사진으로 지정할 수 있어야 한다.
- WHEN 업로드를 시작하면 THEN 위치정보가 제거된 정제 이미지만 서버에 저장되어야 한다.
- IF 갤러리 권한이 거부되면 THEN 권한 설명과 설정 이동 방법을 보여준다.

### US-3. AI 첫 질문

> 사용자로서, 사진을 고른 이유를 스스로 떠올리게 하는 열린 질문을 받고 싶다.

- WHEN 대표 사진 정제가 끝나면 THEN 시스템은 서버 측 Vision 분석을 거쳐 개방형 질문 1개를 생성한다.
- 시스템은 사진 속 관계·장소·감정을 사실처럼 단정하지 않아야 한다.
- IF AI 호출이 실패하면 THEN 재시도 후 기본 질문으로 회고를 계속할 수 있어야 한다.

### US-4. 대화 진행

> 사용자로서, 내 답변을 이어받는 질문을 받고 남은 질문 수를 알고 싶다.

- WHEN 사용자가 답변하면 THEN 시스템은 답변을 저장한 후 핵심 표현을 이어받은 후속 질문을 생성한다.
- 시스템은 남은 질문 수를 표시해야 한다.
- IF 답변이 짧으면 THEN 가벼운 질문을, 구체적이면 한 단계 깊은 질문을 선택해야 한다.
- 사용자는 어떤 질문이든 건너뛸 수 있어야 한다.

### US-5. 회고 종료

> 사용자로서, 원할 때 언제든 회고를 끝내고 지금까지의 내용으로 기록을 만들고 싶다.

- WHEN 질문 3개가 끝나면 THEN 결과 생성으로 이동한다.
- WHEN 사용자가 종료를 선택하면 THEN 저장된 답변으로 결과를 생성한다.
- WHEN 사용자가 2회 연속 건너뛰면 THEN 조기 종료를 제안한다.
- IF 답변이 하나도 없으면 THEN 사진과 날짜로 최소 기록을 저장하고 `draft`로 유지한다.

### US-6. 결과 확인과 수정

> 사용자로서, 하루가 정리된 결과물을 확인하고 필요한 부분을 다듬고 싶다.

- WHEN 회고가 종료되면 THEN 제목·한 줄 회고·기억 문장·필름 카드를 생성한다.
- 기억 문장은 사용자 답변 원문에서 선택되어야 하며 출처를 확인할 수 있어야 한다.
- 시스템은 대화에 없는 감정·사건을 추가하지 않아야 한다.
- 사용자는 제목과 한 줄 회고를 수정할 수 있어야 한다.
- IF 구조화 결과 생성에 실패하면 THEN 최소 기록을 제공하고 재시도를 허용한다.

### US-7. 자동 저장과 아카이브

> 사용자로서, 앱을 중간에 나가더라도 입력한 내용이 유실되지 않고 날짜별로 쌓이길 원한다.

- WHEN 사진 선택을 완료하면 THEN `draft` Film을 저장한다.
- WHEN 질문 또는 답변이 완료되면 THEN 해당 Message를 자동 저장한다.
- WHEN 결과물 저장이 끝나면 THEN Film을 `completed`로 전환한다.
- WHEN 캘린더 날짜를 선택하면 THEN 그날의 정제 사진·대화·결과물을 보여준다.

### US-8. 개인정보와 기록 관리

> 사용자로서, AI에 전달되는 정보와 저장된 기록을 통제하고 싶다.

- WHEN 최초 AI 회고를 시작하면 THEN 외부 AI 처리 고지와 동의를 제공한다.
- 사용자는 개별 기록과 계정을 삭제할 수 있어야 한다.
- WHEN 삭제가 완료되면 THEN 연결된 DB 데이터와 Storage 객체를 더 이상 조회할 수 없어야 한다.

### US-9. 과거 기록 연결 `[확장]`

> 사용자로서, 지금의 나와 과거의 나를 연결해 변화를 발견하고 싶다.

- WHEN 새 기록을 생성하면 THEN 차단되지 않은 기억 포인트와의 유사도를 계산한다.
- IF 연결 신뢰도가 기준 이상이면 THEN 회고당 최대 1회만 연결을 제시한다.
- IF 연결 대상이 없거나 신뢰도가 낮으면 THEN 연결 UI를 노출하지 않는다.
- 사용자는 특정 기록을 “다시 꺼내지 않기”로 설정할 수 있어야 한다.

| 연결 대상 없음 | 연결 기능을 노출하지 않으며 신규 사용자 경험은 연결 없이 완결된다. |
| 연결 제안 무반응 | 해당 연결을 결과물에 포함하지 않는다. |

### 7.2 상태 값

| 대상 | 상태 | 의미 |
|---|---|---|
| Film | draft | 사진 또는 일부 대화만 저장된 미완료 기록 |
| Film | generating | AI 결과물을 생성하고 있는 기록 |
| Film | completed | 결과물과 필름 카드 저장이 완료된 기록 |
| Film | failed | 결과 생성이 실패해 재시도 또는 최소 완료가 필요한 기록 |
| Conversation | active | 질문·답변 진행 중 |
| Conversation | ended | 정상 완료, 사용자 종료 또는 건너뛰기로 종료 |

### 7.3 엔티티 관계

| 부모 | 관계 | 자식 | 비고 |
|---|---|---|---|
| User | 1:N | Film | 사용자 현지 날짜당 최대 1개 |
| Film | 1:N | Photo | `[MVP]` 1~3장, 대표 사진 정확히 1장 |
| Film | 1:1 | Conversation | MVP에서 Film당 회고 세션 1개 |
| Conversation | 1:N | Message | 질문과 답변을 순서대로 저장 |
| Photo | 1:0..1 | PhotoContext | MVP에서는 대표 사진에만 생성 |
| Film | 1:0..N | FilmCard | MVP는 템플릿 1종, 확장 시 여러 결과물 가능 |
| Film | 1:N | MemoryPoint | 확장 기능 |
| Film | 1:N | MemoryLink | 확장 기능 |

### 7.4 핵심 불변식

- `(user_id, record_date)`는 유니크하며 `record_date`는 User.timezone 기준으로 계산한다.
- 모든 Film은 MVP에서 Photo 1~3장을 가지며 대표 사진은 정확히 1장이다.
- 서버에 저장되는 Photo는 정제 완료 상태여야 한다.
- `completed` Film은 제목·한 줄 회고·필름 카드를 가져야 한다. `quote`는 답변이 없으면 null일 수 있다.
- Message.sequence는 Conversation 안에서 유일하고 증가해야 한다.
- PhotoContext는 사용자에게 사실 판단 결과로 직접 노출하지 않는다.
- 차단되거나 삭제된 MemoryPoint는 연결 후보에서 제외한다.

## 10. 기술 스택과 시스템 경계

| 영역 | 기술과 책임 |
|---|---|
| 모바일 클라이언트 | Flutter, `photo_manager` 또는 `image_picker`, `table_calendar`, `RepaintBoundary`; UI, 기기 권한, 이미지 사전 정제, 필름 카드 렌더링 |
| 인증·데이터 | Supabase Auth, PostgreSQL, RLS; 사용자 인증, 기록·대화·상태 저장 |
| 이미지 저장 | Supabase Storage; 사용자별 비공개 경로, 정제 이미지와 필름 카드 저장 |
| 서버 AI Gateway | Supabase Edge Function 등; 인증 검증, API 키 보호, Gemini 호출, JSON 스키마 검증, 재시도·fallback |
| AI | Google Gemini API 무료 티어 사용 가능 모델; 대표 사진 Vision, 질문·결과 생성 |
| 기억 연결 `[확장]` | 임베딩 모델과 Supabase pgvector; 기억 포인트 유사도 검색 |

### 10.1 AI 모델 운영 원칙

- 무료 티어 제공 모델과 할당량은 변경될 수 있으므로 모델명을 환경 설정으로 관리한다.
- 활성 모델과 실제 할당량은 배포 전 Google AI Studio에서 확인한다.
- 모델 변경 시 AI 원칙 회귀 평가를 통과해야 한다.
- 무료 할당량을 초과하면 자동 과금으로 전환하지 않고 fallback 또는 재시도 안내를 제공한다.
- 공식 참고: [Gemini API rate limits](https://ai.google.dev/gemini-api/docs/rate-limits)

> 외부 문서 내용은 라이선스 준수를 위해 요약·재구성했습니다.

### 10.2 데이터 흐름 경계

1. 클라이언트가 사진을 선택하고 리사이즈·압축·EXIF 위치정보 제거를 수행한다.
2. 정제 이미지만 사용자별 비공개 Storage 경로에 업로드한다.
3. 인증된 서버 AI Gateway가 대표 정제 이미지를 읽어 Gemini에 전달한다.
4. 질문과 답변은 각 상호작용 후 PostgreSQL에 저장한다.
5. 구조화 결과를 검증한 뒤 Film에 저장하고 클라이언트가 필름 카드를 렌더링한다.
6. 렌더링한 카드를 비공개 Storage에 저장한 후 Film을 completed로 전환한다.
